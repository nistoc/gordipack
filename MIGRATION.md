Управляется правилом `md-to-sqlite-phased-cutover` (в таблице `rules`).
Принцип: переход **поэтапный и общий для ВСЕХ ролей** — ни одна роль не отключает
md в одиночку, ни одна дормант-роль не пропускается.

## Фазы

### Фаза 0 — Подготовка ✅
- [x] Схема v1 · правила · шаблоны ролей · viewer · init-group прогнан · COORD-промпт дополнен

### Фаза 1 — Двойная запись (запущена)
Роль при каждом тике: читает из md (как раньше) + **дополнительно** пишет в `mezosync.db`.
md остаётся источником правды. Ошибка скрипта → не блокируется, тег `DWERR`.

### Фаза 2 — Двойная запись у ВСЕХ ролей + верификация
Каждая роль при ребёрсе получает в phoenix/launcher инструкцию двойной записи.
COORD верифицирует по каждой роли: дублирует чисто, 0 DWERR, phoenix сохранён в БД.

### Фаза 3 — SQLite = primary
Чтение только из SQLite · md генерируется `export-markdown.py` (read-only для людей) ·
phoenix хранится в таблице `phoenix`.

### Фаза 4 — md отключены
`sync.*.md` больше не пишутся · `phoenix.*.md` генерируется из БД по запросу владельца.

## 🚦 Гейт продвижения фазы (кто и когда переключает)

Перейти **к фазе 3 (отказ от обновления md)** можно ТОЛЬКО когда:

- [ ] ВСЕ роли (у Atlas — 8: coord/core/ing/stud/taxo/rcc/eye/grf) хотя бы раз вошли в двойную запись;
- [ ] у каждой роли есть свежий phoenix-слепок в таблице `phoenix`;
- [ ] 0 открытых DWERR (проверка `check-errors.py`) на протяжении ≥ N тиков;
- [ ] COORD/владелец **явно объявил** продвижение фазы для ВСЕХ (нота + слово владельца).

Пока хоть один пункт не закрыт — md обновляется как обычно. **Роль не бросает md сама.**

## 🔁 Процедура для команды коллег (при обновлении их шаблонов/phoenix)

Старая команда, переходящая на новую версию шаблона, включается ступенчато:

1. **Сначала — двойная запись.** Читает блок «ДВОЙНАЯ ЗАПИСЬ» в своём phoenix,
   пишет в md как раньше + дублирует в SQLite (`write-message.py`/`save-phoenix.py`).
2. **Держит двойную запись,** пока COORD не подтвердит верификацию (фаза 2 для этой роли).
3. **Отказ от обновления md — только после** объявления фазы 3 для всех ролей
   (не раньше и не в одиночку).

Новые группы (через `init-group.py`) стартуют сразу с двойной записью — правило
`md-to-sqlite-phased-cutover` уже в их `rules`.

## 🗄 История

- md-архивы (`coordination/archive/sync.*.<дата>.md`) — **каноничный исторический пласт**,
  человекочитаемый. Задним числом в SQLite **не переносим** (лоссово + дорого, выгоды нет).
- SQLite наращивает историю **вперёд**, с момента старта Фазы 1.
- Полнотекстовые ноты живут в md; SQLite `messages` — сжатые сводки-дубли (пока md = PRIMARY).

## Что НЕ меняется

- Git-коммиты координационных файлов — COORD продолжает коммитить.
- Push — только по живому слову владельца.
- Таймерный механизм (ScheduleWakeup/cron) — в ведении хоста.
- Viewer (`viewer/index.html`) — открывается локально, drag-and-drop `.db`.
