# 🔥 phoenix.README.md — спека «слепок → возрождение»

Контекст чата смертен (компакт, простой, закрытие). Каждая роль держит слепок
`phoenix.<роль>.md`, из которого владелец возрождает роль в НОВОМ чате одной строкой.

## LAUNCHER ≠ REBIRTH

- **LAUNCHER** — одна строка, которую владелец копирует в новый чат:
  `Прочитай <АБС-ПУТЬ>/phoenix.<роль>.md и начни работать по нему.`
- **REBIRTH** — тяжёлый fenced-блок внутри файла: идентичность, порядок чтения, первые действия,
  жёсткие правила. Его исполняет агент, открывший файл.

## Структура слепка (§0–§7)

§0 LAUNCHER → §1 REBIRTH-промпт (fenced) → §2 идентичность и роль → §3 источники правды
(**в порядке чтения**) → §4 ТЕКУЩЕЕ СОСТОЯНИЕ (обновлять чаще всего) → §5 ЧТО ДАЛЬШЕ
(первым делом на пробуждении) → §6 рабочие конвенции + машинные рецепты (пути, гейт, грабли)
→ §7 короткий лог (3–5 вех).

## Дисциплина save

- Обновлять слепок **на каждой вехе** и **перед любой ожидаемой «смертью»** (компакт, конец сессии).
- По команде владельца `phoenix save` — обновить слепок и ЗАКОНЧИТЬ ответ готовым launcher'ом.
- После каждого save — чек возрождаемости: «поднимется ли новый чат ТОЛЬКО из этого файла
  + хвостов sync?»

## Уроки v2

1. **Слепок статичен** — REBIRTH обязан гнать агента в живые ХВОСТЫ sync-файлов; живое состояние
   коллег только там.
2. Карта файлов — в порядке чтения, не «кучей».
3. Машинные рецепты обязательны: точные пути, команды гейта, известные грабли среды.
4. Фиксируй точный HEAD (+ ahead of origin) — ground-truth для возрождённого.
5. Веха — один источник: в слепке ссылка на `[MILESTONE]`-ноту, не пересказ.
